# 核心逻辑与代码思路

本文解释当前 Android/Python 智能车路径规划项目的核心设计，方便在 GitHub 上快速理解“为什么这样建模、代码从哪里看、遇到失败如何排查”。

## 1. 任务抽象

项目把智能车视觉组任务抽象为一个固定 `12 x 16` 栅格问题：

- `P`：小车起点，默认第一列第五行，内部坐标约定为 `(row=5, col=1)`。
- `#`：墙，边界墙不可破坏。
- `B1/T1`、`B2/T2`：编号箱子和对应目标。
- `X`：炸弹，103/105/106 中可推动，撞墙后炸开 3 x 3 范围内的非边界墙。
- 箱子到达同号目标后消失，不再挡路。

Android 内部为了节省字符空间，地图中用：

- `1/2/3/4` 表示 `B1/B2/B3/B4`
- `a/b/c/d` 表示 `T1/T2/T3/T4`
- 导入/导出文本时会自动转换成人可读的 `B1/T1` 格式。

## 2. 规划状态

核心状态由以下部分组成：

```text
state = player_pose + boxes + bombs + dynamic_walls + delivered_mask + recognized_mask
```

含义：

- `player_pose`：小车位置和朝向，转向会计入动作。
- `boxes`：未送达箱子的位置，送达后从状态中移除。
- `bombs`：炸弹位置，爆炸后移除。
- `dynamic_walls`：被炸弹改变后的墙体集合。
- `delivered_mask`：哪些编号箱子已经送达。
- `recognized_mask`：102/103/106 中哪些箱子/目标已经被识别。

## 3. 求解流程

入口在：

- Android：`android_native/app/src/main/java/com/smartcar/planner/planner/NativePlanner.java`
- PC/Python：`planner/solver.py`

整体流程：

1. 校验地图是否合法：边界墙、唯一 `P`、箱子和目标编号对应。
2. 如果关卡需要识别，先规划小车靠近箱子/目标的识别路线。
3. 进入推箱/推炸弹阶段，使用 A* 搜索完整动作序列。
4. 找到结果后，用动作回放校验器验证每一步是否合法。
5. APP 端显示动画、诊断信息，并支持导出错误报告。

## 4. A* 与可达性 BFS

外层 A* 决定“下一步推哪个箱子/炸弹、往哪个方向推”。

内层方向 BFS 判断小车能否走到某个推箱姿态：

```text
push stance = 箱子后方一格 + 正确朝向
```

这样做的原因是推箱子不是普通最短路：小车必须先绕到箱子后方，朝向也必须正确。

严格最短模式中启发式保持 admissible，即不会高估真实代价，因此能保留最短路径保证。STM32 模拟模式则增加节点数、队列、动作数和时间限制，用来测试有限资源下是否还能跑出来。

## 5. 识别阶段

102/103/106 中，箱子和目标需要先识别：

- 小车需要走到目标物前方或可观察位置。
- 识别动作会加入最终动作序列。
- 炸弹不需要识别。

如果识别完成但推箱阶段失败，APP 不再假装是完整方案，而会显示：

```text
仅识别完成
当前仅为识别路径，不是完整推箱方案
```

这类情况会被标记为 `partialRecognitionOnly`，并写入错误报告。

## 6. 炸弹规则

炸弹 `X` 是可推动对象：

- 推向空地：炸弹移动一格。
- 推向墙：炸弹在撞击墙格爆炸。
- 爆炸清除 3 x 3 内非边界墙。
- 边界墙永远保留。

炸弹会改变地图拓扑，因此含炸弹地图不能简单使用固定墙启发式，需要保守处理，避免破坏最短路径正确性。

## 7. 死锁与墙隔断提示

基础死锁剪枝用于减少无效搜索：

- 角落死锁
- 墙边线死锁
- 箱子互堵

Android 端新增了“墙隔断提示”：

- 当识别成功但推箱阶段失败时，非炸弹地图会检查 `Bi -> Ti` 是否被墙/箱子阻隔。
- 如果发现疑似隔断，会在 UI 和错误报告中提示：

```text
B2 到 T2 可能被墙隔断
```

这个提示是排查工具，不替代 A* 的最终判断。

## 8. Android APP 功能

主要文件：

- `MainActivity.java`：界面、按钮、导入地图、导出报告、动画控制。
- `MapEditorView.java`：地图绘制、触摸编辑、动作动画回放。
- `GridMap.java`：地图结构、模板 101-106、合法性检查。
- `MapTextCodec.java`：地图文本导入/导出。
- `ActionReplayValidator.java`：回放校验动作是否合法。
- `NativePlanner.java`：Android 原生规划核心。

APP 当前支持：

- 101-106 模板地图。
- 自定义地图。
- 导入 `.txt` 地图。
- 导出错误报告。
- 横屏动画播放。
- 单步和 x1/x2/x4/x8 快进。
- STM32 strict/relaxed 性能模拟。

## 9. 地图模板

| 地图 | 难度 | 说明 |
| --- | --- | --- |
| 101 | 基础 | 两箱推箱，无识别，无炸弹 |
| 102 | 中等 | 识别 + 两箱推箱 |
| 103 | 较难 | 识别 + 三箱 + 炸弹 |
| 104 | 困难 | 四箱，无炸弹 |
| 105 | 困难 | 双炸弹破墙 |
| 106 | 困难 | 识别 + 三箱 + 炸弹 |

SmokeCore 会同时验证这些地图在 strictShortest 和 stm32Relaxed 模式下可解。

## 10. 错误报告内容

APP 的“导出报告”会保存：

- 当前地图文本
- 选择的关卡和性能模式
- 地图校验结果
- 求解结果
- 动作序列
- 识别顺序
- 诊断计数
- 墙隔断提示
- Java 异常堆栈

这让手机端出现“无解、超时、只完成识别、动作异常”时，可以把报告发回 PC 继续复现。

## 11. STM32F304 思路

STM32F304 不直接运行完整 A*，推荐模式是：

1. PC 或 Android 端完成全局规划。
2. 导出紧凑动作序列。
3. MCU 侧只执行固定动作和小范围局部 BFS。

相关文件：

- `export_stm32.py`
- `embedded/stm32f304/planner_core.c`
- `embedded/stm32f304/planner_core.h`

当前导出设计避免动态内存、递归和 Python 风格的数据结构，更适合小 SRAM 控制器。
